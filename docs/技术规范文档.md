## 0 引言

### 0.1 编写目的

本文档为 **"基于特色创新项目的岭南红橙病虫害智能监测与校本教研一体化平台"**（以下简称"本系统"）的技术规范。文档严格遵循《物体检测项目流程》所定义的**九阶段标准流程**进行组织，每个章节对应流程中的一个阶段，明确该阶段的输入、产出、技术约束、接口规范与验收标准，作为后续编码、训练、测试、部署、运维的统一依据。

### 0.2 项目背景

岭南红橙（大红柑）为国家地理标志产品，其果皮经陈化即为岭南红橙。在岭南红橙现代农业产业园核心种植区域，黄龙病、溃疡病、红蜘蛛、木虱等 12 类高发病虫害严重影响柑果产量与陈皮品质。传统识别依赖人工经验，中老年农户易误判漏判；而现有"云端图像识别"系统对偏远果园网络条件不友好。本系统的目标是研发一套**全本地化运行、零网络依赖、轻量级部署**的智能检测防治系统。

### 2.3 数据标注规范

| 项 | 规范 |
| :--- | :--- |
| 标注工具 | LabelImg / Labelme（统一团队选型） |
| 标注格式 | **YOLO txt**（每行 `class_id cx cy w h`，归一化到 [0,1]） |
| 类别 ID | 0 ~ 11，对应 §1.2 编号 -1 |
| 类别命名 | 中文名映射在 `classes.yaml`，英文 ID 用于训练 |

| 微小虫体标注 | 红蜘蛛 / 木虱若虫定界框最小宽高 ≥ 4 px |
| 群聚虫害 | 每只虫体独立框；超 50 头/图时允许聚类框 + 计数标签 |

### 2.4 数据清洗规则

| 步骤 | 操作 |
| :--- | :--- |
| 去重 | 感知哈希（pHash）检测近似重复样本 |
| 去模糊 | 拉普拉斯方差 < 100 的图剔除 |
| 修正错标 | 农业技术顾问双人交叉复核 ≥ 5% 样本 |
| 类别均衡 | 12 类样本数差距控制在 5× 以内；稀缺类（花蕾蛆等）通过扩增补足 |
| 异常框 | 删除非微虫情况下宽或高 < 4 px 的极小框 |

### 2.5 数据划分

| 子集 | 比例 | 用途 |
| :--- | ---: | :--- |
| 训练集 train | 70% | 模型参数学习 |
| 验证集 val | 20% | 调参与早停 |
| 测试集 test | 10% | 最终评估（不参与训练，含微小目标专项子集） |

## 3 阶段三：数据预处理与增强

### 3.1 推理预处理 Pipeline

| 步骤 | 操作 | 默认值 |
| :--- | :--- | :--- |
| 1. 自适应读图 | 中文路径 `np.fromfile + cv2.imdecode` | — |
| 2. Letterbox | 等比缩放 + 灰边填充 | 输出 640×640 |
| 3. BGR→RGB | 通道顺序转换 | — |
| 4. Normalize | 像素值 /255.0 | — |
| 5. ToTensor / ONNX 输入 | NHWC → NCHW | — |
| 6. 推理 | ONNX Runtime CPU EP | int8 量化 |

> 推理 pipeline 由 ONNX Runtime + Ultralytics 内部完成，**用户无需介入**；GUI 仅暴露 `conf` / `iou` 阈值与物候期选择。

### 3.2 训练数据增强策略

针对田间复杂场景，增强策略在 Ultralytics 默认基础上做了**岭南红橙田间专项调整**：

| 增强 | 默认强度 | 调整后 | 用途 |
| :--- | :--- | :--- | :--- |
| HSV 色彩抖动 | h=0.015, s=0.7, v=0.4 | **h=0.02, s=0.8, v=0.5** | 适应不同光照与季节色差 |
| 水平翻转 | 0.5 | 0.5 | 叶面 / 叶背对称 |
| 垂直翻转 | 0.0 | 0.0 | 不翻转（违反真实拍摄） |
| 旋转 / 平移 / 缩放 | degrees=0, translate=0.1, scale=0.5 | **degrees=15, translate=0.15, scale=0.6** | 多角度拍摄鲁棒 |
| Mosaic | 1.0 | 1.0（最后 10 epoch 关闭） | 多图拼接 |
| MixUp | 0.0 | **0.15** | 复杂叶片背景融合 |
| Copy-Paste | 0.0 | **0.2** | 稀缺类（如花蕾蛆）目标级增强 |
| 高斯模糊 | — | **概率 0.1** | 模拟田间手抖、对焦不准 |

### 3.3 小目标专项增强

针对红蜘蛛、木虱等微型虫害：

| 策略 | 说明 |
| :--- | :--- |
| 多尺度训练 | imgsz 在 [512, 768, 1024] 间随机切换 |
| 锚框微调 | 在 Anchor-Free 基础上调整 head 输出 stride 至 [4, 8, 16, 32]，增加 P2 浅层小目标 head |
| 局部裁剪 | 训练时以微虫为中心做 0.3× 局部裁剪后拼回，强化局部特征学习 |

## 4 阶段四：模型选择

### 4.1 选型对比

| 类型 | 代表模型 | 速度 | 精度 | 部署 | 选用 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| 两阶段 | Faster R-CNN | 慢 | 高 | 复杂 | × |
| 单阶段 | **YOLOv8s** | 快 | 较高 | 极简 | **✓** |
| 单阶段 | YOLOv5s | 快 | 较高 | 简单 | 备选 |
| Transformer | RT-DETR | 中 | 高 | 中等 | 备选 |
| 单阶段 | SSD | 中 | 中 | 简单 | × |

### 4.2 选择 YOLOv8s-XH 的理由

| 维度 | 评估 |
| :--- | :--- |
| 速度 / 精度平衡 | YOLOv8s 在 11.2 M 参数下兼顾 CPU 推理可行性与精度上限 |
| 小目标能力 | Anchor-Free + 多尺度 head，叠加 P2 后红蜘蛛等微虫表现优秀 |
| ONNX 导出 | 官方一行命令导出，CPU 部署友好 |
| 易用性 | Ultralytics 一行 API；本项目算法工程师无学习壁垒 |
| 文档生态 | 中文社区资源丰富，便于廉江本地维护人员后续接管 |
| 模型体积 | 量化后 ≤ 30 MB，满足 NFR-03 极低硬件门槛 |

### 4.3 YOLOv8s-XH 模型定制

`YOLOv8s-XH` 为本项目专属定制模型，基于 YOLOv8s 做以下针对性改造：

| 改造项 | 说明 |
| :--- | :--- |
| 新增 P2 浅层 head | 在 stride=4 处增加输出 head，提升 <16 px 小目标检测能力 |
| 引入 CBAM 注意力 | 在 backbone C2f 模块后插入 CBAM，强化局部特征聚焦 |
| 锚框定制 | 针对红蜘蛛 / 木虱若虫的尺寸分布重新聚类 |
| 类别数 | 12（对应 §1.2） |

### 4.4 模型变体对比（参考）

| 模型 | 参数量 (M) | FLOPs (B) | 选用 |
| :--- | ---: | ---: | :--- |
| yolov8n | 3.2 | 8.7 | 精度不足，备选 |
| **yolov8s-XH** | ~12 | ~30 | **✓ 默认** |
| yolov8m | 25.9 | 78.9 | CPU 偏慢，不选 |

## 7 阶段七：模型优化

### 7.1 优化路径

| 优化类型 | 方法 | 收益 | 本项目是否启用 |
| :--- | :--- | :--- | :--- |
| ONNX 导出 | `yolo export format=onnx` | 跨平台 CPU 部署 | **启用（核心路径）** |
| INT8 量化 | ONNX Runtime PTQ 量化（校准集 ≈ 500 张） | 速度 ↑2~3×，体积 ↓4× | **启用** |
| 半精度 FP16 | ONNX FP16 | 体积 ↓50% | 备选（CPU 收益小） |
| 算子融合 | ORT graph optimization level=All | 速度 ↑10~20% | **启用** |
| 多线程推理 | ORT intra_op_num_threads = 物理核数 | CPU 利用率 ↑ | **启用** |
| 剪枝 / 蒸馏 | 结构化剪枝 + Teacher-Student | 参数 ↓ | 未来 V2 |

### 7.2 ONNX 导出与量化

```bash
# 第一步：导出 ONNX
yolo export model=best.pt format=onnx opset=12 simplify=True imgsz=640

# 第二步：INT8 静态量化（基于 ORT）
python tools/quantize_onnx.py \
    --input  yolov8s_xh_best.onnx \
    --output yolov8s_xh_best_int8.onnx \
    --calibration_dir lianjiang_hongcheng_dataset/images/val \
    --num_samples 500
```

### 7.3 性能预算（实测）

基于 Intel i3-10100 / 8 GB RAM 基准机：

| 模型形态 | 体积 | CPU 单图推理 | 综合响应 |
| :--- | ---: | ---: | ---: |
| YOLOv8s-XH FP32 (.pt) | ~24 MB | ~1400 ms | ~1800 ms |
| YOLOv8s-XH FP32 (.onnx) | ~24 MB | ~900 ms | ~1300 ms |
| **YOLOv8s-XH INT8 (.onnx)** | **~7 MB** | **~600 ms** | **~1000 ms** |

INT8 形态满足 KPI-04（≤ 800 ms）与 KPI-05（≤ 1200 ms）。

### 7.4 量化精度损失控制

| 控制项 | 措施 |
| :--- | :--- |
| 校准集多样性 | 必须覆盖 12 类、5 个物候期、多光照 |
| 精度回归 | 量化后 mAP 损失需 ≤ 1.5 个百分点；否则回退至 FP32 |
| 保留 FP32 备份 | 客户端可在设置中切换至 FP32 高精度模式（牺牲速度） |

| 处方 | 说明 | 示例 |

- 药剂名（仅允许国家登记的低毒低残留药剂，如螺螨酯、阿维菌素等）
- 稀释倍数（`2000~3000 倍液` 或 `每亩 50 mL`）
- **安全间隔期 PHI**：**必须突出显示**（如 `安全间隔期：21 天`）
- 混用禁忌（如 `请勿与碱性农药混用`）

### 8.16 教学教研一体化扩展模块

> **定位说明**：本模块是在现有离线检测系统基础上，**扩展教学与教研内容生成能力**，面向高职实训、教师教研场景。模块不改变第 1~7 阶段的检测主链路，作为 8.x 部署能力的增量功能交付。所有生成内容**仅作为教研与实训辅助材料，须经教师审核确认后用于教学**，本系统不承担全自动教学决策职能。

#### 8.16.1 模块目标与原则

| 原则 | 说明 |
| :--- | :--- |
| 默认完全离线 | 与主系统一致，**默认**零网络依赖；大模型默认本地推理。云端模式为用户可选、非默认、需主动联网授权（见 8.16.3.3） |
| 辅助而非替代 | 生成内容为草稿，**教师审核后方可用于教学**，系统保留"待审核 / 已审核"状态 |
| 知识可更新 | 法规与处方依赖本地知识库，支持人工审核后定期更新（见 8.16.5） |
| 低配可用 | 模型规格按实训电脑算力弹性选择，不绑定单一大模型 |
| 不阻塞主链路 | 内容生成为可选附加步骤，模型缺失或推理失败自动降级，不影响检测 / 处方 / 台账 |

#### 8.16.2 本地边缘端病虫害智能识别子模块（8.16.A）

复用第 8.7 智能检测分类模块能力，面向教学场景封装为标准化流程：

1. **多源图像输入**：本地图片导入 / UVC 摄像头 / 数字显微镜 / 本地视频帧。
2. **图像预处理**：letterbox 缩放、归一化（复用 §3.1 推理预处理 Pipeline）。
3. **YOLOv8 本地推理**：按 §8.2 档位策略自动选择 PT / ONNX FP32 / ONNX INT8 后端。
4. **检测结果可视化**：彩色定界框 + 中文标签 + 严重程度分级 + 群聚虫害计数。
5. **性能指标与测试环境说明**（验收口径，避免无条件刚性指标）：

| 指标 | 测试环境 | 期望 |
| :--- | :--- | :--- |
| 单张图片平均推理时延 | 指定测试环境（i5 级 CPU / 16 GB / ONNX Runtime / 640×640 输入） | ≤ 200 ms |
| 整体检测响应时间 | 普通教学实训笔记本环境（含读图 + 推理 + 处方匹配） | ≤ 1 s |

> 说明：上述 ≤ 200 ms 为**指定测试环境下的平均推理时延**，非全部设备的无条件刚性指标。低配设备以 §1.4 KPI-04（CPU 单图 ≤ 800 ms）/ KPI-05（综合响应 ≤ 1.2 s）为兜底验收门槛，INT8 量化形态用于保障低配机的响应。

#### 8.16.3 离线教学/实训内容生成子模块（8.16.B）

| 能力 | 说明 |
| :--- | :--- |
| 本地大模型调用 | 支持本地部署的**轻量化中文或中英双语指令模型（如 3B~8B 量化模型）**，按实训电脑算力选择规格；不绑定特定模型型号 |
| 检测结果与知识库联合注入 | 将检测结果（病虫害类别、严重程度、计数、物候期）与本地知识库处方（§8.8）作为上下文注入提示词，约束生成内容不偏离已审核知识 |
| 教学案例 Markdown 生成 | 输出结构化教学案例（病情描述 + 诊断依据 + 三位一体防治 + PHI 提示），Markdown 格式 |
| 学生实训指导生成 | 生成实训任务书 / 操作步骤 / 思考题，供学生实训使用 |
| 教师审核机制 | 生成内容默认进入"待审核"状态，教师可编辑、批注、确认；**仅"已审核"内容允许导出用于教学** |
| 导出 PDF / MD | 审核通过后导出 PDF（复用 §8.9 reportlab 通路，含 CJK 字体）或 Markdown |

> **模型规格约束**：高职实训电脑算力有限，规范**不绑定 8B 及以上模型**；以 3B~8B 量化模型为可用区间，优先保障离线、低配机上的稳定推理与中文教研文本质量。具体选型在《模型部署说明》中按交付硬件确定。

##### 8.16.3.1 功能需求（FR-G）

| 编号 | 需求 | 优先级 | 阶段 |
| :--- | :--- | :--- | :--- |
| FR-G1 | 检测结果页新增「生成教研内容」入口 | 高 | V1.1 |
| FR-G2 | 两种模板：①《岭南红橙实时防害教学案例》②《学生农技实训指导意见》 | 高 | V1.1 |
| FR-G3 | 自动把检测结构化结果 + 知识库处方注入 Prompt，约束生成不偏离已审核知识 | 高 | V1.1 |
| FR-G4 | 本地大模型离线推理，生成 Markdown 草稿 | 高 | V1.1 |
| FR-G5 | 生成内容进入「待审核」状态，教师可编辑 / 批注 / 确认 | 高 | V1.1 |
| FR-G6 | **仅"已审核"内容允许导出**，复用现有 PDF / Markdown 通路（§8.9） | 高 | V1.1 |
| FR-G7 | 模型缺失 / 推理失败时降级为"模板填充"，不阻断检测主流程 | 中 | V1.1 |
| FR-G8 | 注入物联网温湿度等环境参数做情境化生成 | 低 | **V2.0（留接口）** |

> **闭环逻辑**：① 前端识别（摄像头 / 图片 → 本地 YOLOv8 → 结构化检测结果）→ ② 大模型推理（检测结果 + 知识库处方〔+ V2.0 环境参数〕拼成 Prompt，本地/可选云端推理）→ ③ 内容生成（教学案例 / 实训指导草稿 → 教师审核 → 导出）。生成环节为**可选附加步骤**，失败不影响检测、处方与台账。

##### 8.16.3.2 生成文档结构

**《岭南红橙实时防害教学案例》**：病情概述 → 检测依据（类别 / 置信度 / 计数 / 严重程度）→ 物候期背景 → 三位一体防治（含 **PHI** 提示）→ 教学要点 / 思考题。

**《学生农技实训指导意见》**：实训目标 → 操作步骤 → 安全注意事项（PHI / 用药规范）→ 观察记录表 → 考核要点。

##### 8.16.3.3 大模型调用方案（双模式）

为兼顾"完全离线"定位与生成质量，采用**双模式**架构：

| 模式 | 说明 | 默认 | 网络 |
| :--- | :--- | :--- | :--- |
| **本地模式** | 本地部署 3B~8B 量化模型（如 llama.cpp / GGUF），本机离线推理 | ✅ **默认** | 零依赖 |
| **云端模式（可选）** | 用户自行配置云端大模型 API Key（如 DeepSeek / 豆包），切换后联网调用 | 否 | 需联网 |

> **重要约束**：系统**默认本地离线**，"完全离线、零网络依赖"卖点以本地模式为准；云端模式为**用户可选配置项、非默认、需用户主动联网授权**，规范与申报材料中须明确标注，避免与离线定位冲突。云端模式下检测图像与结果是否外发由用户知情同意后决定。

#### 8.16.4 内容生成数据流

```
检测结果（类别/严重程度/计数/物候期）
    +
本地知识库处方（§8.8，已人工审核）
    + 〔V2.0 留接口：物联网温湿度等环境参数〕
    ↓ 联合注入提示词（约束生成）
大模型推理（默认本地 3B~8B 量化离线；可选云端 API）
    ↓ 生成草稿（教学案例 / 实训指导）
进入"待审核"状态
    ↓ 教师审核 / 编辑 / 确认
"已审核"内容 → 导出 PDF / Markdown 用于教学
```

#### 8.16.5 知识库与法规更新规范

为避免离线系统对"最新法规"作出无法保证的承诺，本模块的合规表述统一为：

> 系统依据**预置并经人工审核的本地法规 / 处方知识库**生成防治与教学建议，**支持知识库定期更新**（更新周期见 §9.4）。系统不声称实时同步最新法规；生成内容须以教师 / 农业技术顾问审核为准。

### 13.2 关键设计决策（ADR）

| ID | 决策 | 理由 |
| :--- | :--- | :--- |
| ADR-01 | 推理用 ONNX Runtime 而非 PyTorch | 用户端零 Python 依赖；CPU INT8 加速友好 |
| ADR-02 | GUI 用 PyQt5（非 PySide6） | 中文生态资源更多；适老化控件成熟 |
| ADR-03 | 知识库用 SQLite | 嵌入式、免配置、跨平台、单文件分发 |
| ADR-04 | 推理用 QThread 而非 multiprocessing | ORT Session 跨进程序列化困难 |
| ADR-05 | 中文路径 `np.fromfile + imdecode` | OpenCV 对非 ASCII 路径不友好 |
| ADR-06 | Python 锁定 3.11.x | 平衡新特性与生态兼容 |
| ADR-07 | 完全离线，无遥测 | 农户隐私 + 偏远果园无网 |
| ADR-08 | Excel 用 openpyxl 而非 win32com | 跨平台、用户端无需 MS Office |
| ADR-09 | INT8 静态量化（PTQ）而非 QAT | 训练成本低，精度损失可控 |
